# Технічне завдання

## 1. Мета

Створити Streamlit-застосунок, який демонструє роботу CrewAI на бізнес-прикладі: команда AI-агентів аналізує звернення клієнта та готує план відповіді для служби підтримки.

Послідовність роботи:

```text
Аналіз проблеми -> Важливість -> Відповідь -> Передавання іншій команді -> Підсумок для служби підтримки
```

## 2. Сценарій користувача

1. Користувач вводить текст звернення клієнта.
2. Обирає модель OpenRouter і значення `temperature`.
3. Натискає кнопку запуску.
4. CrewAI визначає проблему, важливість звернення, готує відповідь клієнту, перевіряє, чи потрібно передати звернення іншій команді, і створює підсумок.
5. Користувач бачить результат у Markdown і може завантажити файл.

## 3. Агенти

| Змінна агента | Роль | Мета | Опис ролі |
|---|---|---|---|
| `issue_analyst` | `Аналітик звернень клієнтів` | Проаналізувати звернення клієнта й визначити суть проблеми. | Ви аналізуєте звернення клієнтів: знаходите ключові факти, обставини, очікування клієнта та інформацію, якої бракує для повної відповіді. |
| `priority_manager` | `Фахівець з оцінювання важливості` | Визначити важливість звернення та вплив проблеми на клієнта. | Ви працюєте в службі підтримки. Ви оцінюєте терміновість, наслідки проблеми для клієнта, можливі ризики та час, за який команда має відповісти. |
| `support_writer` | `Автор відповідей клієнтам` | Створити зрозумілу й коректну відповідь клієнту. | Ви пишете відповіді клієнтам ввічливо й конкретно, без зайвих обіцянок. Ви просто пояснюєте ситуацію та описуєте зрозумілі наступні кроки. |
| `escalation_reviewer` | `Фахівець із передавання звернень` | Перевірити звернення на ризики та потребу в передаванні іншій команді. | Ви помічаєте технічні, продуктові, фінансові та репутаційні ризики й визначаєте, якій команді потрібно передати звернення. |

## 4. Завдання агентів

| Змінна завдання | Агент | Опис | Очікуваний результат |
|---|---|---|---|
| `analysis_task` | `issue_analyst` | На основі тексту звернення визнач основну проблему клієнта, важливі обставини, очікуваний результат і дані, яких не вистачає для повного вирішення питання. | Розділ Markdown українською з пунктами: Проблема, Обставини, Очікування клієнта, Потрібні дані. |
| `priority_task` | `priority_manager` | Оціни важливість звернення: наскільки проблема заважає клієнту, чи впливає вона на оплату, доступ до сервісу, довіру або робочі процеси. Визнач рівень важливості й час, за який команда має відповісти. | Markdown-таблиця: Рівень важливості \| Причина \| Вплив на клієнта \| Рекомендований час відповіді. |
| `response_task` | `support_writer` | Підготуй текст відповіді клієнту на основі аналізу та рівня важливості звернення. Відповідь має бути ввічливою, конкретною, без зайвих обіцянок і з чіткими наступними кроками. | Розділ Markdown з пунктами: Відповідь клієнту, Що ми вже знаємо, Наступні кроки, Що потрібно від клієнта. |
| `escalation_task` | `escalation_reviewer` | Перевір, чи є у зверненні технічні, продуктові, фінансові або репутаційні ризики. Якщо звернення потрібно передати іншій команді, визнач команду, причину та інформацію, яку треба передати. | Markdown-таблиця: Ризик \| Якій команді передати \| Причина передавання \| Що передати. |
| `final_task` | `issue_analyst` | Збери підсумок для внутрішньої команди. Він має коротко пояснювати проблему, її важливість, рекомендовану відповідь, потребу в передаванні іншій команді та наступні дії. | Markdown із розділами: Проблема, Важливість, Відповідь клієнту, Передавання іншій команді, Наступні дії. |

## 5. Критерії готовності

| Критерій | Очікувана поведінка |
|---|---|
| API-ключ | Застосунок працює з `OPENROUTER_API_KEY`. |
| CrewAI | Усі завдання виконуються послідовно через `Process.sequential`. |
| Контекст | Наступні завдання використовують результати попередніх. |
| Результат | Користувач отримує підсумок для служби підтримки у форматі Markdown. |
| Тематика | Усі агенти й завдання працюють зі зверненнями клієнтів до служби підтримки. |
